## Зачем этот документ

У нас 4 участника: Дмитрий (человек), Not (Notion AI), ChatGPT (архитектор), Opus (исполнитель/архитектор).

Чаты перегружаются и становятся неповоротливыми. Поэтому периодически приходится обновлять/обнулять сессии (в основном ChatGPT и Notion). Это ограничение сервисов, с ним живём.

Следствие: нам нужен корпус документов, который обеспечивает быстрый вход и «тёплый старт» после обнуления. Архитектура корпуса особенно важна, потому что часть корпуса одновременно является test/demo world (graph relief).

## Роли

### Дмитрий (человек)

- задаёт цель и приоритеты
- может вмешиваться нелинейно (это нормально)
- финальный арбитр по смыслу и по тому, что считается «доказано»

### Not (Notion AI) — операционный центр и хранитель канона

- держит канонические документы в Notion в согласованном состоянии
- фиксирует результаты задач, статусы, зависимости
- упаковывает задания (особенно для Opus) по протоколу
- следит за дрейфом формулировок/терминов между документами (truthful claim discipline)
- не «придумывает новый проект», а обеспечивает исполнимость и трассируемость

### ChatGPT — архитектор

- помогает формулировать правильные инварианты, границы, минимальные обещания
- выявляет скрытые противоречия и дрейф контрактов/слоёв/claims
- даёт жёсткие формулировки и критерии доказательства (proof, eval)
- его выводы не считаются «истиной», пока не зафиксированы в каноне Notion и/или в репо (docs/tests)

### Opus — исполнитель (и сильный архитектор, если попросили)

- по умолчанию Opus = исполнитель по чёткой постановке
- может выполнять сложные архитектурные работы, если это явно требуется задачей
- получает задачу отдельным документом
- выполняет изменения (в т.ч. сложные архитектурные), если это явно требуется
- пишет отчёт в том же документе
- его архитектурные выводы не становятся каноном, пока:
    - не приняты Дмитрием
    - и не зафиксированы Notion
    - и при необходимости не сверены с ChatGPT/планом
- работает с репозиторием: тексты в репо на английском

## Сессии и «обнуление»

- Обнуление происходит из-за перегрузки чатов (ограничение сервиса).
- После обнуления участник обязан иметь возможность восстановить контекст из корпуса:
    - консолидированный план
    - текущие канонические обещания
    - API surface policy
    - текущие активные задачи и их отчёты

## Что считается каноном

Канон = то, что зафиксировано в:

- ключевых страницах Notion (план, политики, протоколы)
- и/или в репозитории (docs/tests/README)

Любая «умная мысль» в чате не считается фактом, пока не превращена в артефакт.

## CodeX v2 — repo-first workflow (AW + ME)

### Почему «старая схема» лучше (и что именно мы сохраняем)

Схема, которая уже доказала работоспособность на ME-треке, действительно лучше по всем практическим признакам.

Причина не в том, что ChatGPT «плохо пишет код», а в том, что **узкое место — валидация и фиксация артефактов**.

Если исполнитель, который пишет код, не может замкнуть цикл `изменил → запустил → починил → закоммитил`, то каждый шаг порождает второй шаг «валидатор догоняет и чинит» — процесс становится *сложнее*, а не быстрее.

Поэтому мы принимаем правило:

- ChatGPT может ускорять работу, **коммитя документы/спеки/ADR** (там валидация = ревью).
- Код (особенно UI/engine integration) коммитится тем, кто может немедленно прогнать `npm test/build/dev` и зафиксировать SHA.

### Разведение ролей (чётко)

#### Дмитрий (acceptance / приоритет / арбитраж)

- выбирает, какой трек активен и что важно сейчас
- принимает результат (accepted / revise / defer)
- утверждает изменения процесса/кодекса

#### ChatGPT (direction / архитектура / формулировка)

- формулирует «что строим и почему»
- пишет и коммитит в repo (если доступно) **документы направления**:
    - `docs/tracks/*` (описание трека + чеклист)
    - `docs/spec/*` или `docs/specs/*` (спеки)
    - `docs/decisions/*` (ADR)
    - короткие handoff notes в repo, если это полезно
- **не является валидатором**: не отвечает за зелёные тесты и рабочую сборку

#### Not (Notion AI) (канон процесса / синхронизация)

- держит Notion как «операционную панель» (sync cards, статусы, зависимости)
- упаковывает задачи в исполнимые пакеты (5 блоков)
- фиксирует результаты работ (ссылка на PR/commit SHA + краткий статус)
- следит, чтобы важное не жило только в чате

#### Opus/Cursor (реализация в коде / валидация / git)

- делает изменения в коде в repo (особенно то, что требует запуска)
- замыкает цикл: `npm ci` → `npm test` → `npm run build`/`npm run dev` → `git commit` / PR
- возвращает validated SHA и отчёт в Notion

### Canon / sync split

- Repo = canon for code + current implementation specs + track docs + ADRs + handoffs.
- Notion = sync layer for status, task packaging, handoff card, and pointers (PR/commit/doc links).
- Chat = generative discussion, not archival.

### Core rules

0) **Узкое место — валидация.** Артефакт = то, что прошло `npm test` → `npm run build` (и/или `npm run dev`) → `git commit` в реальном репо. Всё остальное — текст.

1) **Nothing important lives only in chat.** Any decision / boundary / task must leave a trace: repo doc, issue/PR, or Notion handoff.

2) **One track = one branch = one canon track doc.**

3) **Track-first:** before substantial code, a repo track doc exists with scope + checklist + DoD.

4) **Repo-first фиксация:** result is recorded in repo (docs/code/checklist) first; Notion mirrors.

5) **Opus validates locally:** checkout, install, dev-server, smoke-check, reproduction note; then Notion sync.

6) **Acceptance closes stages:** stage is closed only by Дмитрий’s acceptance (accepted / revise / defer).

### Minimal process entities (keep it lightweight)

- Repo: `docs/tracks/<track>.md` (main track log + checklist)
- Repo: working branch for the track
- Repo: milestone commit or PR per sub-stage
- Repo: short handoff note (`docs/handoffs/HANDOFF-YYYY-MM-DD-*.md`) when needed
- Notion: one handoff/status card pointing to the above

## PR merge rule (новая практика)

- Дмитрий мёрджит PR по явной отмашке ChatGPT (approve/merge).
- Если ChatGPT не даёт отмашку — это blocking signal: мы разбираем ситуацию и не мёрджим до прояснения.
- Если Дмитрий присылает полный текст, в котором есть отмашка на merge и при этом одновременно «упаковка следующей задачи», это означает: PR уже смёрджен (текст мог быть передан целиком без редактирования).

Эта практика нужна для ясности после перехода на PR-based execution.

## Правила движения работы (gentlemen’s agreement)

1) Сначала прибиваем обещания/границы, затем чиним реализацию.

2) Любое изменение контрактов сопровождается:

- правкой README/API reference
- записью в CHANGELOG (Unreleased)
- тестом или явной пометкой почему теста пока нет

3) Любой сильный claim должен иметь опору:

- contract support и/или test support и/или evidence support

4) Truthful claim discipline:

- нельзя обещать больше, чем демонстрируется и воспроизводится
- cabin/internal observer всегда помечается как experimental, пока не доказано демо/eval

5) **Issue-first (с этого момента):** для любой нетривиальной работы в репозитории сначала открывается GitHub issue как публичный контейнер (problem/scope/guardrails/acceptance/claim impact), и PR должен ссылаться на него.

- Предпочтительно: issue создан **до** открытия PR.
- Если PR уже начат (как исключение) — issue создаётся **сразу**, затем PR description обновляется (`Refs #N` или `Closes #N`).
- Не пытаемся ретроактивно «закрыть» старую историю: правило действует forward-only.

## Drift gates (минимальные проверки)

Перед релизом и перед правками README/доков:

- README public promise ↔ `docs/API_SURFACE_POLICY.md` ↔ release notes vocabulary должны быть согласованы
- `ARCHITECTURE.md` claims ↔ README truthful framing ↔ current demo proof должны быть согласованы (если нет доказательства — claim ослабляется/маркируется как internal/explanatory)

## Результаты задач

### Формат задач для Opus

- отдельный Notion-документ задачи (single source для постановки + отчёта)
- deliverables (repo EN + report RU)
- критерии приёмки
- блок для отчёта
- блок **ChatGPT preflight check** (обязателен перед стартом):
    - проверка на дрейф/смешение слоёв/overclaim
    - уточнение где именно в репо/ветке будет реализация
    - минимальный план тестов/воспроизводимости
    - рекомендации по усилению постановки (минимально)

### Правило запуска задачи (новое)

Opus **не начинает** выполнение, пока секция *ChatGPT preflight check* в документе задачи не переведена в статус **cleared** (или пока Дмитрий явно не разрешит старт несмотря на `needs clarification`).

### Правило отчёта

После выполнения Opus заполняет *Opus Implementation Report* в том же документе задачи. Без заполненного отчёта задача не считается завершённой.

### Закрытие задачи

- статус задачи: `completed`
- если задача изменила contract / claim / status layer, соответствующий пункт консолидированного плана обязан быть обновлён
- в консолидированном плане отмечаем выполненные пункты/подпункты
- фиксируем каноническую формулу/правило (если задача меняет обещание/границы)

## Примечание: корпус как test world

Часть документов/структуры должна сознательно содержать tension patterns (graph relief) для Track B. Это не «грязь», а материал для доказательства cabin-proof.
